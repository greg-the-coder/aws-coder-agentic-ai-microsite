# Agentic AI Observability Dashboard: Recommendations

## Context

This document proposes a CloudWatch-based observability dashboard that demonstrates how the AWS + Coder microsite (https://partner.coderdemo.io/aws+coder) was developed and published using Coder Agents in governed workspaces. It synthesizes activity across four observable dimensions — Coder Agent execution, LLM interactions, GitHub operations, and AWS infrastructure — into a single pane of glass using Amazon CloudWatch Generative AI Observability.

The dashboard serves as both a **proof-of-concept** for the microsite's "Modernize" and "Multiply" messaging and a **reference implementation** for customers evaluating governed agentic development.

## Bedrock Guardrails Recommendations

Based on the observed agent behavior in this session, the following Bedrock Guardrails should be deployed to limit and alert on erroneous or malicious activity:

### 1. Content Filters (Guardrail Policy)

| Filter | Threshold | Rationale |
|--------|-----------|-----------|
| **Denied Topics** | Block: PII generation, credential exposure | Agent should never generate or expose secrets, API keys, or PII in published content |
| **Word Filters** | Alert: internal hostnames, private IPs, account IDs | Prevent accidental exposure of infrastructure details in public-facing output |
| **Sensitive Information** | Block: SSN, credit card, AWS access keys | Redact any sensitive patterns before content reaches S3 |

### 2. Token & Cost Guardrails

| Guardrail | Configuration | Alert Action |
|-----------|---------------|--------------|
| **Max tokens per invocation** | 32,000 output tokens | CloudWatch Alarm → SNS notification |
| **Max tokens per session** | 2,000,000 total | Hard stop + alert to workspace owner |
| **Cost ceiling per session** | $25 estimated spend | Throttle to smaller model + alert |
| **Anomaly detection on token rate** | 3σ above baseline | Auto-pause agent + Slack notification |

### 3. Tool Execution Guardrails

| Guardrail | Policy | Enforcement |
|-----------|--------|-------------|
| **Allowed AWS services** | S3, CloudFront, CloudWatch only | Agent Firewall network policy (deny all except allowlisted endpoints) |
| **Blocked commands** | `rm -rf /`, `aws iam`, `aws ec2 terminate`, destructive operations | Agent Firewall command blocklist |
| **Git push targets** | Only `github.com/greg-the-coder/*` | Network policy restricting git remote URLs |
| **File write scope** | Only `/home/coder/microsite/**` | Workspace-level filesystem isolation |
| **S3 bucket scope** | Only `aws-coder-microsites-coderdemo` | IAM policy condition on resource ARN |

### 4. Output Validation Guardrails

| Guardrail | Check | Action on Failure |
|-----------|-------|-------------------|
| **HTML validation** | Published HTML must be well-formed | Block S3 upload, alert |
| **Content size limit** | No single file > 500KB | Block upload, alert |
| **No external scripts** | HTML must not reference external JS (XSS prevention) | Block upload, alert |
| **Brand compliance** | Scan for competitor mentions, off-brand language | Alert for human review |
| **Link validation** | All href targets must resolve (no broken links) | Warning alert |

### 5. Behavioral Anomaly Detection

| Signal | Baseline (this session) | Alert Threshold |
|--------|------------------------|-----------------|
| Commands per minute | ~2–3 | > 20/min (possible runaway loop) |
| S3 uploads per hour | ~4 | > 30/hr (possible infinite deploy loop) |
| Git force pushes | 1 (initial only) | > 1 after initial setup |
| CloudFront invalidations/hr | ~3 | > 15/hr (cost concern + possible abuse) |
| Unique AWS services accessed | 3 (S3, CF, STS) | > 6 (scope creep alert) |
| Failed API calls | < 5% | > 20% (possible permission escalation attempts) |

## Implementation Path

### Phase 1: Instrumentation (1–2 days)

1. **Install CloudWatch Agent** on Coder workspace template with OpenTelemetry collector
2. **Enable Bedrock model invocation logging** to CloudWatch Logs
3. **Configure GitHub webhook** → EventBridge → CloudWatch Logs
4. **Emit custom EMF metrics** from Coder Agent wrapper script for tool usage tracking

### Phase 2: Dashboard (1 day)

1. Create CloudWatch Dashboard with 4 panel groups (Agent, LLM, GitHub, Infra)
2. Import prebuilt Bedrock GenAI observability widgets from CloudWatch console
3. Add Logs Insights query widgets for top-N token consumers and cost breakdowns
4. Configure X-Ray trace groups for agent session correlation

### Phase 3: Guardrails & Alerting (1–2 days)

1. Create Bedrock Guardrail with content filters + denied topics
2. Deploy Agent Firewall policies in Coder workspace template
3. Create CloudWatch Alarms for token ceiling, cost anomaly, and behavioral baselines
4. Configure SNS topic → Slack/email for alert routing
5. Add CloudWatch Anomaly Detection on `TotalTokens` and `ShellCommandsExecuted`

### Phase 4: Iterate

1. Correlate dashboard signals with deployment success/failure outcomes
2. Tune guardrail thresholds based on observed agent patterns
3. Add CloudWatch Synthetics canary to validate published microsite health
4. Integrate with Coder's AI Gateway audit log for unified LLM usage view

## Summary

This dashboard demonstrates that the development of a co-branded microsite — from research through iterative refinement and deployment — is fully observable using AWS-native tooling. The combination of:

- **CloudWatch GenAI Observability** for LLM tracing and token economics
- **Coder Agent Firewall** for execution-layer governance
- **Bedrock Guardrails** for content and cost safety rails
- **CloudTrail + X-Ray** for audit and dependency mapping

...provides the enterprise control plane necessary to run autonomous AI agents at scale with confidence. This is the "Modernize" and "Multiply" story in practice — not theoretical, but demonstrated through the very creation of this site.
